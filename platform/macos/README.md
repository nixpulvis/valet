# ValetAutoFill — macOS AutoFill credential provider

Valet's macOS password AutoFill integration. An
`ASCredentialProviderExtension` ships inside `ValetAutoFill.app` and talks to
the `valetd` daemon over a Unix socket via the `valetd::ffi` C surface.

`ValetAutoFill.app` (bundle id `com.nixpulvis.valet.autofill`) is a minimal
host whose only job is to satisfy macOS's requirement that a credential
provider `.appex` live inside a `.app`. When `valet-gui` gets a real macOS
`.app` bundle (`com.nixpulvis.valet`), the `.appex` moves into its
`Contents/PlugIns/` and `ValetAutoFill.app` is retired.

## Requirements

- macOS 14 or later
- Xcode Command Line Tools (`xcode-select --install`)
- Rust with the `aarch64-apple-darwin` target: `rustup target add aarch64-apple-darwin`
- No Xcode project, no GUI tooling — this builds entirely from the command line

## Build

    make                      # arm64, real daemon, ad-hoc signed
    make BUILD_MODE=stub      # in-process stub, no daemon required
    make ARCH=x86_64          # Intel
    make universal            # lipo'd arm64 + x86_64

Artifacts land in `build/`. The staticlib comes from the `valetd` workspace
member (`../valetd/`); `valet.h` and `valetd.h` are generated by `valet-build`
into `../target/<triple>/include/`. Signing defaults to ad-hoc (`-`). Override
via `make SIGN='Developer ID Application: …' DEVELOPMENT_TEAM=XXXXXXXXXX`.

## Install and enable

    make install     # copies build/ValetAutoFill.app to /Applications/ and launches
                     # it once so pluginkit registers the extension and the App
                     # Group container is provisioned

Then:

1. Open **System Settings → General → AutoFill & Passwords**.
2. Toggle **ValetAutoFill** on.
3. Start the daemon (see below).
4. Click the AutoFill / key icon in a password field in Safari. Pick
   **Valet**; the extension's unlock prompt appears, then the credential
   list.

Inspect with:

    make verify
    log stream --predicate 'subsystem == "com.nixpulvis.valet.autofill"'

## Run the daemon

The AutoFill extension is sandboxed, so the socket has to live inside the
App Group shared container rather than in `$XDG_DATA_HOME`. The App and the
extension both declare `com.apple.security.application-groups = [group.com.nixpulvis.valet]`;
macOS creates the container on first launch of an entitled process.

Start `valetd` pointing at that same path:

    VALET_SOCKET="$HOME/Library/Group Containers/group.com.nixpulvis.valet/valet.sock" \
        cargo run -p valetd

Leave it running in a terminal. The daemon caches unlocked user keys for
5 minutes of inactivity (see `IDLE_TIMEOUT` in `valetd/src/bin/valetd.rs`);
the extension's first AutoFill invocation after that window drops you back
into the unlock prompt.

Database path is independent of the socket: set `VALET_DB` to override, or
let it default to `$XDG_DATA_HOME/valet/valet.sqlite`. Use
`cargo run --bin valet -- register <username>` first if you do not yet have
a registered user.

### Stub mode

`make BUILD_MODE=stub install` links an in-process `StubClient` that returns
two canned records (ycombinator.com / example.com). No daemon, no database,
no real unlock. Useful for iterating on the Swift UI in isolation.
